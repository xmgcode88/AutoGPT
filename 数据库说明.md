  # Studio（http://localhost:8082）与 PostgreSQL（postgresql://…:5432）的区别、联系与“为什么看起来像部署了两个数据库”

  ## 1. 先一句话结论

  - http://localhost:8082/ 是 Supabase Studio（Web 管理后台），它是一个网页应用，用来可视化管理项目、表、权限、用户、存
    储等。
  - postgresql://localhost:5432/... 是 PostgreSQL 数据库服务，是真正存储数据、执行 SQL 的数据库引擎。
  - 你并没有“为了 Studio 再部署一个独立数据库”；通常只有一个主 Postgres（supabase-db）。你看到的“多个数据库”更多是：
      - 同一个 Postgres 容器里包含多个 数据库（database）/ schema（例如 postgres、_supabase、auth、storage 等），不是多
        套数据库服务器；
      - 或者还有 平台自身的业务数据库/其他服务数据库（如果你同时启了 AutoGPT 平台全套服务），那是“另一个项目/应用”的数据
        库需求，与 Supabase 的内部库不同。

  ———

  ## 2. http://localhost:8082/ 是什么？

  ### 2.1 它是“前台/控制台”，不是数据库

  - Studio 是一个 Web UI（Next.js/Node 运行的网页服务）。
  - 你用浏览器打开 http://localhost:8082/，实际上是在访问 supabase-studio 这个容器暴露的 HTTP 服务。

  ### 2.2 Studio 能做什么？

  常见能力包括：

  - 查看/编辑数据表（类似轻量级数据库管理工具）
  - 管理认证用户（Auth）、策略（RLS）、角色
  - 管理存储桶（Storage）
  - 查看项目配置、API 文档、SQL 编辑器等（取决于你启动的 Supabase 组件是否完整）

  Studio 的这些功能最终都要“落到后端服务/数据库”上，自己不存储业务数据。

  ———

  ## 3. postgresql://…:5432 是什么？

  ### 3.1 它是真正的数据库引擎

  - 5432 是 PostgreSQL 默认端口。
  - 你连接 localhost:5432，就是在连接 supabase-db 容器里运行的 Postgres 实例。

  ### 3.2 PostgreSQL 负责什么？

  - 真实数据持久化（表、索引、行数据）
  - SQL 查询与事务
  - 权限/角色/RLS（Supabase 很多“安全能力”依赖 Postgres 原生特性）

  ———

  ## 4. Studio 和 PostgreSQL 的联系是什么？

  可以理解为一条链路：

  1. 你在浏览器打开 Studio：http://localhost:8082
  2. Studio 需要读取“项目元信息/表结构/权限/用户”等
  3. Studio 会请求 Supabase 的后端服务（例如 Kong 网关、Meta 服务等，具体由 compose 里服务决定）
  4. 后端服务最终会连到 Postgres（supabase-db）读取或修改数据/元数据

  所以：

  - Studio 是“操作界面”
  - Postgres 是“最终数据与规则的来源”
  - 中间还会有网关/元数据服务等“翻译层/接口层”

  ———

  ## 5. 为什么你会感觉“部署了两个数据库”？

  常见误解来源有三类：

  ### 5.1 同一个 Postgres 里有多个“数据库/Schema”

  Supabase 的 Postgres 里往往会有：

  - 多个 schema：例如 auth、storage、extensions、public 等
  - 甚至多个 database：例如 _supabase（给某些内部组件用）、postgres（默认库）等

  这不是“多部署了一套数据库服务器”，而是 一个 Postgres 实例内部的多命名空间/多数据库。

  ### 5.2 你同时启动了“平台业务数据库”和“Supabase 数据库”

  在 AutoGPT 平台这种工程里，可能还存在平台后端自身的数据库（用于平台业务表、任务、用户资料等）。这会让人感觉：

  - Supabase 有一套 Postgres（supabase-db）
  - 平台也需要一套 Postgres/或同一套但不同 schema/不同连接串

  如果你的 compose 把平台服务也一起拉起来了，那么“数据库需求”会更复杂；但就你这次命令（只启动 kong auth db studio meta）
  而言，核心就是 Supabase 的 Postgres。

  ### 5.3 Studio 看起来像“一个数据库产品”

  Studio UI 里能直接编辑表数据，很像“数据库本体”，但它只是 UI；真正的数据读写仍由 Postgres/后端服务完成。

  ———

  ## 6. 那为什么要有 Studio 这个“8082 服务”？直接用 Postgres 不行吗？

  直接用 Postgres 当然可以（你可以用 DBeaver/psql/PGAdmin 连接 localhost:5432 管理数据）。但 Studio 的价值在于：

  - 把 Supabase 的概念（Auth、RLS、Storage、API、项目设置）做成一体化 UI
  - 以“项目视角”串起数据库、权限、认证等组件
  - 降低日常管理门槛（尤其是非 DBA/非后端同学）

  换句话说：

  - Postgres 是底座
  - Studio 是把 Supabase 的“产品层能力”可视化的控制台

  ———

  ## 7. 你现在这套本地启动里，“到底有几个数据库”？

  从“数据库服务器实例”的角度：

  - 通常 只有 1 个 Postgres 实例：supabase-db（端口 5432）

  从“实例内的 database/schema”的角度：

  - 可能有多个 database / schema（这是 Postgres 内部结构，不等于多部署一套）

  从“工程整体”的角度（如果你将来启动平台全家桶）：

  - 可能还会有平台自己的数据库需求（取决于你启动哪些服务、是否复用同一个 Postgres、是否用不同 schema）。
